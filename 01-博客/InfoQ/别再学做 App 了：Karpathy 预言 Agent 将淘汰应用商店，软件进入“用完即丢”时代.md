# 别再学做 App 了：Karpathy 预言 Agent 将淘汰应用商店，软件进入“用完即丢”时代

## 文档信息
- 来源：https://www.infoq.cn/article/4LPuZzDH4CXh5G5h5N4G?utm_source=rss&utm_medium=article

## 摘要
**1) 一句话总结**
Andrej Karpathy 预言，随着 AI Agent 能够通过自然语言按需、即时地生成个性化工具，传统应用商店将被淘汰，软件将进入“按需生成、用完即丢”的新时代。

**2) 核心要点**
*   **按需生成软件：** Karpathy 预测未来用户只需描述需求，AI 即可现场生成专属 App，未来该过程可能仅需 1 分钟。
*   **硬核实验数据：** Karpathy 耗时 1 小时利用 AI (Claude) 生成了约 300 行代码，定制了一个专属的有氧训练仪表盘（目标是 8 周内将静息心率从 50 降至 45）。
*   **效率大幅提升：** AI 成功反向解析了 Woodway 跑步机的云端 API 并完成了数据清洗，此类工作在两年前大约需要耗费 10 个小时。
*   **生态基础设施滞后：** 当前生成速度未能达到“1分钟”的瓶颈在于生态未准备好，99% 的产品和服务仍缺乏 AI 原生的 API 或 CLI。
*   **软件形态转变：** 软件正从固定的“标准化产品”转变为“按需生成的临时工具”，自然语言（英语）成为新的编程接口。
*   **企业服务重构：** 传统的 SaaS 采购和 UI 界面可能被 Agent 动态生成的工具和编排所替代。
*   **核心竞争力转移：** 随着大模型日益标准化，行业竞争焦点将转向基础设施、算力以及 Agent 工作流的部署速度。
*   **应用概念淡化：** Karpathy 认为，当软件变得极度廉价且可随时生成时，普通用户无需再具备“App”的概念，一切交互将由 LLM 代理完成。

**3) 风险与缺口**
*   **代码准确性风险：** 在实际生成中，AI 仍会出现搞错公制/英制单位、日历日期匹配出错等 bug，需要人工介入指出并修复。
*   **API 接口缺口：** 现有设备和服务过度依赖面向人类的前端界面和 .html/.css 文档，缺乏能被 Agent 直接调用的底层 API。
*   **安全与审核缺口：** 网友指出，应用商店承担着信任、审核和安全机制，完全淘汰存在风险，未来可能需要经过验证的基础层来保障安全。
*   **需求理解风险：** 反对者指出，非专业用户缺乏领域词汇，仅凭随意的语音提示，LLM 难以准确破解并弄清楚客户真正的需求。

## 正文
如果软件可以由 AI 现场生成，那应用商店会不会消失？

知名 AI 工程师、神经网络布道者 Andrej Karpathy 最近发布的一篇长帖引发了热议。他的核心观点非常明确：**以后可能不用专门去应用商店“找” App 了，你只要说清楚想干嘛，AI 现场就能给你做一个——而且未来可能只要 1 分钟。**

### 1 小时手搓专属仪表盘：Karpathy 的硬核实验

为了展示高度定制化软件时代的模样，Karpathy 拿自己做了一个实验。

他最近打算在有氧训练上更加自律，计划用 8 周时间，把静息心率从 50 降到 45（主要通过 Zone 2 有氧训练和每周 1 次的 HIIT）。一般人遇到这种需求，大概率会去应用商店搜索“有氧实验追踪器”之类的 App。但 Karpathy 另辟蹊径，他没有下载现成的 App，而是直接用 AI Agent 临时做了一个专属的追踪仪表盘。

仅仅 1 小时后，他通过 Vibe Coding 就搞定了这个量身定制的 Web 前端界面。在这个过程中，AI（Claude）反向解析了 Woodway 跑步机的云端 API，拉取原始数据，并完成了清洗、过滤和调试。

虽然过程并非完美无缺——比如 AI 搞错了公制和英制单位，日历的日期匹配也出了 bug 需要人工指出并修复——但这大约 300 行代码，LLM Agent 几秒钟就能生成。在两年前，完成这些大概需要 10 个小时。

### 为什么还要 1 小时？生态阻力与 AI 原生 API 的缺失

尽管 1 小时的效率已经很高，但 Karpathy 并不满足，他兴奋地思考：**这件事本来应该只需要 1 分钟。**

他分析指出，时间并没有花在“理解需求”上，而是主要消耗在了弥补基础设施的缺陷上。现在的最大阻力不在模型能力，而是生态还没准备好。他犀利地吐槽了当前的行业现状：

> “99% 的产品和服务仍然没有 AI 原生的 CLI。99% 的产品和服务仍然维护着 .html/.css 的文档，好像我不会第一时间把内容复制粘贴给我的 Agent 去完成任务似的。他们在网页上给出一串操作说明……现在都 2026 年了，为什么还要我手动操作？要么系统自动完成，要么交给我的 Agent。”

他认为，整个行业需要重构一套服务体系。以他的跑步机为例，它本质上是一个将物理状态转化为数字信息的传感器。它不应该只维护一个面向人类的前端界面，让 Agent 去反向工程；而是应该提供一个能被 Agent 轻松调用的 API 或 CLI。

如果设备本身提供 Agent 可直接调用的 API，如果常见功能有成熟的技能库，如果 AI 已经掌握了个人的长期背景数据，那么理论上只剩下“描述需求 + 自动拼装”。只需说一句：“嗨，能帮我在接下来 8 周追踪有氧训练吗？”应用就能在简短问答后自动搭建完成。

### 当语言成为编程接口：软件即当下时刻

其实，这一切早有伏笔。Karpathy 在 2023 年写下的名言——“目前最热门的新编程语言是英语”，至今还挂在他的主页置顶。

当自然语言本身就是编程接口，软件就不一定非要先被做成一排排固定的产品等人去下载。它可以在对话里被拼出来，在具体场景中存在，用完就消失。从这篇帖子中，我们可以提炼出关于软件未来的几个核心预判：

*   **软件形态的转变：** 软件正在从“标准化产品”，变成“按需生成的临时工具”，变得更短暂、更个性化。
*   **机器间直接对话：** 硬件和软件需要提供 AI 原生 API，让机器和机器直接对话，而不是依赖人为操作界面。
*   **企业服务重构：** 企业可能不再大量购买 SaaS，而是动态生成工具。传统的 UI 也可能被 Agent 编排所替代。
*   **核心竞争力转移：** 真正的竞争力不再是谁“懂 AI”，而是谁能最快把 AI 部署成实际可用的系统。大模型正变得越来越标准化，竞争将转向基础设施、算力和 Agent 工作流的部署速度。

科技媒体“AIM Network”对此形容道：“我们过去收集软件的方式就像收集书一样：精挑细选、下载安装、定期更新，然后慢慢被遗忘。未来的流程可能不再是‘下载—安装—配置—适配’，而是‘描述—生成—使用—丢弃’。”

软件将不再是一个被打包、上架、定价的产品，而更像一段即时生成的服务，即**“软件即当下时刻”**。

### 网友吵翻天：应用商店会消失吗？奶奶能自己做 App 吗？

Karpathy 的观点引发了网友的激烈讨论。

**应用商店会消失吗？**
很多人认为，应用商店不会突然消失，而是会进化成“应用商店 2.0”或“3.0”版本。因为它承担着信任、审核和安全机制。未来更可能出现的画面是混合模式：不再售卖一堆细碎的 App，而是提供可靠的底座（经过验证的基础层），让 Agent 在上面自由拼装 AI 即时生成的个性化扩展。要消失的未必是软件本身，而是“软件必须以固定产品形态存在”这个概念。

**普通人真的需要自己做 App 吗？**
有质疑者认为 Karpathy 是在以 AI 从业者的视角自卖自夸：“你是个厨师，但并非每个人都想成为或愿意成为厨师。”还有人锐评：“你奶奶会自己做 App 吗？”

对此，Karpathy 回应称，人们还在用“软件稀缺”的旧思维看问题。当软件变得极度廉价、可随时生成时，传统意义上的“应用”本身可能都会失去存在意义。“奶奶根本没必要懂什么 App，甚至不用知道有 App 这回事；这些该由她的 LLM 代理知道。”

**AI 真的懂客户需求吗？**
反对者依然指出：“构建软件最困难的部分是弄清楚客户真正想要什么。但你声称一个 LLM 能从一位连问题领域词汇都不具备的人的随意语音提示中破解它？”

支持者则坚信，未来的 AI 就是最好的设计师和需求分析师。一旦生成速度足够快，分发逻辑、SaaS 结构甚至企业 IT 预算的分配，都会随之发生连锁反应。

对于未来 App 的形态和“用完即丢”的软件时代，你怎么看？
