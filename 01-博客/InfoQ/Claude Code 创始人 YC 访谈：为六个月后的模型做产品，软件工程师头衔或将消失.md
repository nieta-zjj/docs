# Claude Code 创始人 YC 访谈：为六个月后的模型做产品，软件工程师头衔或将消失

## 文档信息
- 来源：https://www.infoq.cn/article/Ki4t4mNFY92hisCVaxwB?utm_source=rss&utm_medium=article

## 摘要
**1) 一句话总结**
Claude Code 创始人 Boris Cherny 在 YC 访谈中指出，AI 正在彻底改变编程方式，开发者应为六个月后的模型构建产品，未来“软件工程师”头衔或将消失并演变为综合型构建者（Builder）。

**2) 核心要点**
*   **生产力显著提升：** 自 Claude Code 推出后，Anthropic 内部人均工程产出提升了 150%，部分员工 70%–100% 的代码由 Claude 编写。
*   **极简的产品形态：** Claude Code 最初仅为学习 API 而开发的低成本终端程序，因其极简约束和自然体验，意外从原型变成了最终产品形态。
*   **顺应潜在需求开发：** 核心功能均从用户现有行为中“长”出来，例如用户习惯让模型先规划，团队仅耗时 30 分钟便开发上线了广受欢迎的 Plan mode。
*   **为六个月后的模型构建：** 开发者应预测并适应极短的代码生命周期（Claude Code 六个月前的代码已几乎全部重写），避免围绕当前模型做 PMF。
*   **不要和模型对赌：** 坚信更通用的方法最终会赢过特化的方法，下一代模型原生具备的能力会直接取代当前人工添加的辅助代码。
*   **多 Agent 协作（Agent Swarm）：** 通过“不相关的上下文窗口”隔离 Agent，避免上下文污染；Claude 的 Plugins 功能几乎完全由 Agent Swarm 在一个周末内无人工干预自主完成。
*   **无 IDE 的未来：** 创始人 Boris 每天能合并 20 个 PR 且不再手写代码、已卸载 IDE；团队仅用 10 天（几乎 100% 由 AI 编写）推出了 co-work 图形界面，让销售、财务等非技术人员也能使用 Agent。

**3) 风险/不足**
*   **技术债风险：** 为弥补当前模型不足而过度添加的非模型本体代码（脚手架/Scaffolding），在下一代新模型发布后会直接变成技术债。
*   **产品淘汰风险：** 如果仅围绕当前模型的能力构建产品市场契合度（PMF），很快会被下一代模型的能力抹平。
*   **认知滞后风险：** Agent 的能力边界每几个月就会重写，工程师若不具备“新手心态”并持续重置认知，其对模型能力的判断会迅速过时。
*   **上下文污染风险：** 在多 Agent 协作中，如果不进行上下文隔离，Agent 之间或历史记录会产生相互污染。
*   **指令冗余风险：** 当 CLAUDE.md 等系统指令文件变得过长时，会导致模型偏离正轨（建议直接删除重来，用最少指令拉回正轨）。

## 正文
“我们会开始看到‘软件工程师’这个头衔慢慢消失。可能会变成 builder、product manager，或者头衔还保留，但只是一个遗留符号。因为大家做的工作不再只是写代码：软件工程师还会写 spec、还会跟用户沟通。”

放出这话的，正是 Claude Code 的创始人 Boris Cherny。

在近期 Y Combinator 的一场圆桌访谈中，Boris 一人对阵四位 YC 高管（总裁兼 CEO Garry Tan、合伙人 Harj Taggar、Diana Hu 以及 Jared Friedman），分享了关于 AI 编程、产品构建以及未来开发者形态的硬核观点。

在他看来，编程正在被“解决”。在 Anthropic，很多人已经 70%–100% 用 Claude 写代码，IDE 的存在感正在下降。写代码这件事，正在从“核心能力”变成“默认能力”。

YC 总裁 Garry Tan 坦言：“谢谢你做了 Claude Code。它让我连续三周没睡好，感觉像给人装上了火箭助推器。”这并非客套，自 Claude Code 推出后，Anthropic 的人均工程产出提升了 150%。Boris 表示，这种 100% 级别的生产力提升在过去是闻所未闻的。

这场对话最核心的价值，在于揭示了 Boris 如何把一个终端里的小聊天程序，迭代成今天能调工具、会规划、甚至主动找人沟通的开发 Agent。

### 核心观点提炼

*   **代码的保质期只有几个月：** Claude Code 被反复重写，六个月前的代码几乎全部消失；重构不是例外，而是常态。
*   **Plan mode 未来可能自动进入甚至消失：** 其本质只是在 Prompt 里加一句“先别写代码”，最终可能一发 Prompt 就能完成所有规划与执行。
*   **不要和模型对赌：** 加很多脚手架（Scaffolding）也许能多拿 10% 的效果，但下一代模型可能直接“白送”；所有非模型能力最终都会变成技术债。
*   **功能是从用户行为里“长”出来的：** Plan mode、CLAUDE.md、co-work 都源于用户已经在做的事，产品只是把它们收拢进来。不要教育用户改变行为，而是顺着他们已经发生的行为去放大。
*   **Agent 的能力边界每几个月重写一次：** 你对“它能不能做”的判断很快会过时，工程师必须不断重置认知。多 Agent 协作（并行 Sub-agent、上下文隔离）是能力放大的关键变量。
*   **迭代速度本身就是护城河：** Claude Code 可以一天做 20 个原型，快速试错比“设计完美方案”更重要。
*   **新时代最重要的能力是“新手心态”：** 能承认自己错、能丢掉旧经验、能从第一性原理重新思考，比资历更重要。

### 意外的终点：极简终端成就 Claude Code

Claude Code 的诞生充满了偶然。最初，Anthropic 内部认为通往安全 AGI 的路径之一是通过编程能力。当时市场上有一种强烈的“产品能力悬空感”——模型已经具备了编程潜力，但还没有人做出能将其“吃干榨净”的产品。

为了学习如何使用 Anthropic 的 API，Boris 随手写了一个很小的终端程序。选择终端仅仅是因为成本最低、不用做 UI。他尝试让模型读取文件、控制 Mac 播放音乐，结果模型展现出了强烈的“使用工具的意愿”。

“最不可思议的是：我们到现在居然还在用终端。终端本来只是起点，我没想到它最后会变成终点。”Boris 回忆道。

原型做出的第二天，同事就已经开始在电脑上用它写代码了。这种极简、优雅的终端形态，像一个很好的设计约束，让开发体验变得像玩一样，自然而然地在内部火了起来。

### 顺应潜在需求：从用户行为中“长”出功能

Boris 强调，产品里最大的原则就是**潜在需求（Latent Demand）**。人们只会去做他们本来就在做的事情，产品要做的就是让他们“本来就想做的事”变得更容易。

Claude Code 的许多核心功能都源于此：
*   **CLAUDE.md：** 早期用户习惯自己写 Markdown 文件让模型读，团队顺势将其产品化。Boris 建议，如果 CLAUDE.md 变得太长，最好的处理方式是直接删掉重来，用最少的指令把模型拉回正轨。
*   **Plan mode：** 源于用户经常在聊天窗口里对 Claude 说“先帮我想方案，但先别写代码”。团队花 30 分钟写出这个功能，当晚发布，立刻大受欢迎。

随着模型能力的提升，Plan mode 的生命周期可能有限。以前用户需要在规划前后全程盯着模型（Babysit），现在只需要在规划前确认方向。未来，可能只需一个 Prompt，Claude 就能自己想清楚并执行完毕。

### 认知重置与多 Agent 协作

“Agent 真正能做什么，会随着每一代模型变化。”Boris 坦言，他经常被团队里激进的用法震惊，比如让 Claude Code 自己写个小工具来分析内存泄漏。工程师必须不断“重置认知”，因为大脑往往还停留在六个月前。

在协作形态上，**Agent 拓扑（Agent Topologies）** 正在成为全新的领域。其核心思路之一是“不相关的上下文窗口（Uncorrelated Context Windows）”：让多个 Agent 各自拥有干净的上下文，避免被彼此或历史记录污染。

投入更多上下文本质上是一种 Test-time Compute。通过合适的拓扑结构让 Agent 并行沟通，它们就能构建更庞大的系统。例如，Claude 的 Plugins 功能，几乎完全是由一个 Agent Swarm（智能体集群）在一个周末里自己“跑”出来的，几乎没有人工干预。

### 给技术创始人的建议：为六个月后的模型构建

面对 AI 洪流，Boris 给出了两条反直觉但至关重要的建议：

**第一，不要为今天的模型构建，要为六个月后的模型构建。**
如果只围绕当前模型做 PMF（产品市场契合度），很快会被下一代能力抹平。开发者需要不断摸清模型的能力边界，为即将到来的能力提前做准备。

**第二，不要和模型对赌（Never bet against the model）。**
在开发中，团队经常面临权衡：是花精力写非模型本体的代码（脚手架）来提升 10% 的体验，还是等几个月让新模型原生解决？Boris 的答案是：假设你做的脚手架最终都会变成技术债。更通用的方法最终会赢过特化的方法。

正因如此，Claude Code 的代码库被反复重写，六个月前的代码现在几乎荡然无存。代码的保质期只有几个月，顶尖的创始人应该预期并适应这种极短的生命周期。

### 1000倍工程师与无 IDE 的未来

AI 工具的加持正在重塑团队结构。拥有清晰产品愿景的创始人，借助 Claude Code 能实现 50 倍的效率放大。在 Anthropic 内部，不仅技术员工每天使用，甚至一半的销售团队、财务团队和设计师也在用。

为了满足非技术用户的潜在需求，团队仅用 10 天时间（且几乎 100% 由 Claude Code 编写）就推出了带有 GUI 外壳的 co-work，让更多人能安全、便捷地使用 Agent 能力。

“自从 Opus 4.5 之后，我把 IDE 都卸了，不再手写任何一行代码，每天能落 20 个 PR。”Boris 预测，沿着指数曲线推演，编程会逐渐对每个人都被“解决”。

未来，写代码将不再需要 IDE。工程师将变成通才，跨越产品、设计和业务。当写代码的门槛被彻底抹平，“软件工程师”的头衔或许将成为历史，取而代之的，是纯粹的 Builder。
